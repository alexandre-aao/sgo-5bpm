# SGO 5º BPM — Sistema de Gestão Operacional

Aplicativo full-stack para a Seção de Planejamento (P3) do 5º Batalhão de Polícia Militar. Nasceu como "Pauta de Eventos" (gestão de eventos e diárias) e cresceu para cobrir também o Cartão Programa de patrulhamento diário, cadastros de referência (bairros, pessoal) e relatórios operacionais. Todo o domínio é em português — nomes de variáveis, campos de dados e textos de interface usam a terminologia da corporação (Ofício, diária, viatura, guarnição, Fiscal de Operações etc.).

## Stack e arquitetura

- **Backend:** Node.js + Express, arquivo único `server.js`. Sem TypeScript, sem framework de rotas além do Express puro.
- **Frontend:** HTML/CSS/JS vanilla, sem build step, sem framework (nada de React/Vue). SPA por abas: todo o HTML das telas fica em `public/index.html` dentro de `<section class="tab-content">`, escondido/mostrado via classe `active`.
- **Banco de dados:** Supabase (Postgres). `server.js` acessa via `@supabase/supabase-js` com a chave `service_role` (bypassa RLS; a autorização é toda feita pelo próprio app — `autenticar`/`exigirP3`). Para preservar a lógica de negócio já escrita em JS puro (filter/map/reduce) sem reescrever tudo em SQL, existe um shim `readDB()`/`writeDB()` em `server.js` que busca/grava cada tabela inteira e monta o mesmo formato de objeto que o código já esperava do antigo `data/db.json`. `writeDB(db, tabelas)` recebe a lista explícita de tabelas alteradas por aquela escrita — nunca sincroniza as 8 tabelas inteiras à toa. Mesmo assim, `writeDB` faz upsert+diff da tabela inteira (não só da linha alterada), o que pode causar "lost update" entre duas escritas concorrentes na mesma tabela; nas quatro tabelas de maior concorrência (`cartoes`, `escalas`, `eventos`, `operacoes`) as rotas de escrita usam os helpers pontuais `writeRow(tabela, row)`/`deleteRow(tabela, id)` em vez de `writeDB`, evitando o read-modify-write da tabela inteira — ao adicionar uma nova rota de escrita nessas quatro tabelas, usar `writeRow`/`deleteRow`, não `writeDB`. As demais tabelas continuam no padrão `writeDB` original. Rotas quentes (`autenticar`, `/api/login`) usam consultas pontuais (`buscarSessaoPorToken`, `buscarUsuarioPorLogin`) em vez do shim. **Rotas GET de tabela única** (`/api/eventos`, `/api/operacoes`, `/api/pessoal`, `/api/escalas`, `/api/alocacoes`, `/api/viaturas`, `/api/bairros-coordenadas`) usam o helper `readTabela(tabela, filtros = {})` (`SELECT *` de UMA tabela, `.eq()` por filtro; valor vazio/undefined é ignorado) em vez de `readDB()` — evita baixar as outras 9 tabelas à toa (`readDB` fazia ~11 SELECTs por request; o boot do frontend eram ~77). Filtros por igualdade vão no objeto (`readTabela('escalas', { operacao_id })`); filtros por array (categoria de `pessoal`) e ordenação (`localeCompare`) continuam em JS. `config` é linha única → `buscarConfig()`. **Ao criar uma GET nova de tabela única, usar `readTabela`, não `readDB`.** As rotas **agregadoras** (que cruzam várias tabelas: `dashboard-resumo`, `estatisticas*`, `relatorio*`, `planejador-diarias`, `diarias-calendario`) continuam com `readDB()`. Respostas são comprimidas com `compression()` (gzip) e os estáticos têm `Cache-Control: max-age=3600` (assets) / `no-cache` (index.html). **Cache-busting:** as referências a `app.js`/`style.css` no `index.html` recebem `?v=<ASSET_VERSION>` — como o `index.html` é `no-cache`, o HTML novo (com a nova versão) chega na hora e o navegador refetcha os assets em vez de servir os de até 1h atrás do cache. `ASSET_VERSION` = `VERCEL_GIT_COMMIT_SHA` (8 chars, muda a cada deploy na Vercel automaticamente) com fallback pro `mtime` do `app.js` em dev local. O `index.html` **não é mais servido pelo `express.static`** (que agora tem `index: false`): existe uma rota `app.get(['/', '/index.html'])` que serve uma cópia do HTML pré-lida no boot com o `?v=` já injetado (`INDEX_HTML`) — ao adicionar novo `<script>`/`<link>` versionável no `index.html`, incluir a substituição correspondente nesse bloco do `server.js`. Schema completo em `supabase/schema.sql` — mudanças de schema são SQL aditivo (`ALTER TABLE ... ADD COLUMN IF NOT EXISTS`) rodado manualmente pelo usuário no SQL Editor do Supabase (não há migration runner automático), e a tabela nova/coluna nova entra em `TABELAS`/`CHAVE_PRIMARIA` no topo do `server.js`.
- **Sem dependências de frontend via bundler** (npm só serve o backend). CDNs usados: Lucide (ícones, versão fixada em `index.html` — ex: `lucide@1.24.0` — nunca usar `@latest`, evita quebra de ícones por update não controlado da lib), Google Fonts, **Leaflet** (mapa, aba Mapa).
- **Deploy:** Vercel, função serverless única (`server.js` via `@vercel/node`, ver `vercel.json`). Projeto conectado ao repositório GitHub `alexandre-aao/sgo-5bpm` — todo `git push` na branch `main` dispara deploy automático em produção (`https://sgo-5bpm.vercel.app`). Variáveis de ambiente `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` configuradas no painel da Vercel (Project Settings → Environment Variables), **não** commitadas — localmente ficam em `.env.local` (git-ignorado) e são carregadas também via `.claude/launch.json` pro preview server.
- **Não é monorepo.** Só existe este projeto neste diretório.

## Rodando o projeto

```bash
npm install
npm start        # ou: node server.js
```

Sobe em `http://localhost:3005` por padrão (ver `.claude/launch.json`; `autoPort: true` deixa o harness de preview escolher outra porta livre se a 3005 já estiver em uso por outra sessão). Precisa de `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` no ambiente — já configuradas em `.claude/launch.json` para o preview server.

**Sempre reiniciar o servidor** depois de editar `server.js` — não há hot reload. Se o usuário reportar "erro no servidor" em algo que você acabou de mudar, o primeiro suspeito é sempre um servidor real (`npm start` do próprio usuário, fora do preview do Claude Code) rodando código antigo sem reiniciar.

## Modelo de dados (Supabase / Postgres)

- **`eventos`** — Eventos_Pauta: eventos **civis, sem diária** (Show, Futebol, Ato Público, Religioso, Cultural, Evento Junino, Outros). **Não têm mais escala nominal nem diária** — isso migrou para `operacoes` (ver abaixo). Campos-chave: `data_inicio`/`data_termino`, `bairro` (select alimentado por `bairros_coordenadas` + opção "Outro" com texto livre), `num_os_manual` e `num_sei` (texto livre, preenchidos manualmente — **não existe mais geração automática de código de OS nem status de OS/Kanban, foram removidos deliberadamente**).
- **`operacoes`** — registro **único** de operação policial (planejamento → execução, sem duplicar registro como fazia a antiga `missoes_planejadas`). Campos: os mesmos de `eventos` mais `tipo_operacao` (Ostensiva, Saturação, Cerco, Blitz, Cumprimento de Mandado, Reforço, Outras — em `TIPOS_OPERACAO` no `server.js`), `situacao` (`Planejada` | `Executada` — só muda via "Marcar como Executada", sem efeito colateral em diária), `qtd_diarias_estimada` (reserva de cota enquanto ainda não há escala), `tipo_recorrencia` (`diaria`|`fim_de_semana`|`dia_unico`, opcional). Diária da operação = `diariaDaOperacao(op, escalas)`: soma real das escalas se houver, senão a estimativa. É onde a diária vive agora. A antiga `missoes_planejadas` foi migrada para cá e a tabela ficou **órfã no banco** (dados migrados, sem `DROP`, sem código lendo — não recriar dependência nela).
- **`alocacoes`** — Alocacao_Policiamento: efetivo/viaturas alocados (modalidade, qtd_policiais, qtd_viaturas). Vinculada a **UM `evento_id` OU UM `operacao_id`** (nunca aos dois, nunca a nenhum — constraint `alocacoes_um_vinculo`).
- **`escalas`** — Escala_Diarias: militares escalados por **operação** (`operacao_id`, não mais `evento_id`). Regra de negócio fixa: **`total_diarias = qtd_aparicoes * 2`**. Sem trava por `situacao` — pode escalar em operação Planejada ou Executada.
- **`usuarios`** — login, senha (hash scrypt), `role` (`P3` | `Adjunto` | `Oficial`), nome. Conta de sistema (login), diferente de `pessoal`.
- **`pessoal`** — Cadastro de Pessoal: `nome`, `nome_guerra` (nome de guerra, texto livre — populado nos 244 registros; usado na busca do autocomplete de escala), `matricula` (RE, texto livre opcional), `subunidade` (lista fechada em `SUBUNIDADES_PESSOAL` no `server.js`: `PCS`, `1ª Companhia`, `2ª Companhia`, `3ª Companhia` — só descritivo, não tem lógica de negócio associada ainda), `posto_graduacao` (lista fechada da hierarquia PMRN em `POSTOS_GRADUACAO` no `server.js`), `tipo` (`Praça` | `Oficial`, derivado automaticamente do posto), `categorias[]` (`Adjunto`, `Fiscal de Operações`, `Oficial de Operações`, `Oficial de Sobreaviso`, `Executor` — uma pessoa pode ter mais de uma, ou nenhuma). Alimenta os seletores de Fiscal/Adjunto/Sobreaviso do Cartão Programa. `categorias` é opcional (decisão deliberada — o batalhão tem efetivo geral cadastrado só como registro, sem papel operacional ainda; não exigir categoria de volta sem confirmar com o usuário). O filtro "Sem categoria" na tela é só client-side (`app.js`), a API não tem esse filtro nativamente. Na importação de 2026-07 (ver "Pendências"), `categorias` foi inferida a partir da "Função Atual" do relatório SGEPM só nos casos inequívocos (função literal "Adjunto" → categoria Adjunto; funções de execução de patrulha — Comandante de VTR, Patrulheiro, Motorista de VTR, Guarda — → categoria Executor); funções de comando/plantão (Comandante do BPM, Delta 07 etc.) e administrativas ficaram deliberadamente sem categoria por ambiguidade — não inferir esse grupo automaticamente numa limpeza futura sem confirmar com o usuário, pois afeta diretamente os seletores reais de Fiscal/Oficial de Operações/Sobreaviso do Cartão Programa.
- **`bairros_coordenadas`** — `nome_bairro`, `latitude`, `longitude`. Seed inicial: 7 bairros da Zona Sul de Natal (coordenadas aproximadas de centróide, não geocodificação precisa). Alimenta o select de Bairro em Novo Evento e os marcadores do Mapa.
- **`cartoes`** — Cartão Programa: um por data (`data`), com `viaturas[]` (prefixo, setor, companhia, categoria, comandante, observação) e `itens[]` de roteiro (horário início/fim, local, atividade). Campos de cabeçalho `fiscal`, `adjunto`, `oficial_sobreaviso` guardam o **nome** da pessoa selecionada (não um id — mantém compatibilidade com cartões antigos que tinham texto livre). Também serve de **template**: `is_template` (bool), `nome_template`, `tipo_periodo` (`semana` | `fim_de_semana`), `qtd_viaturas_base` (5/6/7), `origem_template_id` (rastreia de qual template um cartão do dia foi clonado). Templates têm `data: null` e nunca aparecem nas listagens normais por data.
- **`sessoes`** — tokens de autenticação ativos (expiram em 12h).
- **`config`** — `cota_mensal_diarias`.

### Regras de negócio importantes

- **Diária:** `qtd_aparicoes * 2`, sempre. Não mexer nessa fórmula sem confirmar com o usuário. **Diária é das `operacoes`, não dos `eventos`** (eventos civis não geram diária). No Planejador: `total_consumido` = soma real das escalas das operações que já têm escala; `total_planejado` = soma de `qtd_diarias_estimada` só das operações **sem** escala. Nunca a mesma operação nos dois — evita contagem dupla. Ver `diariaDaOperacao` no `server.js`.
- **Não existe mais status de evento nem Kanban de Ordem de Serviço** — foi removido deliberadamente (era `status_os`/`codigo_os`/Kanban no Dashboard). Não reintroduzir sem pedido explícito do usuário. No lugar: `num_os_manual` e `num_sei`, campos de texto livre preenchidos manualmente. O Dashboard alerta quando um evento próximo/em andamento está sem Número da OS e/ou Número SEI informado (calculado, não é status manual).
- **Praça x Oficial no Cartão Programa:** se o Fiscal de Operações selecionado for classificado como `Praça` (posto/graduação até Subtenente) e o campo Oficial de Sobreaviso estiver vazio, o sistema mostra um alerta **não-bloqueante** no painel de Alertas de Conflito do Cartão — não impede salvar.
- **Templates do Cartão Programa:** P3 cria templates nomeados (período + qtd. de viaturas base) e monta viaturas/roteiros com o comandante em branco. No dia, quem cria o Cartão (Adjunto/P3) **escolhe manualmente** o tipo (Dia Útil = `semana` / Fim de Semana = `fim_de_semana`) no select `#sugestao-tipo-periodo` + qtd. de viaturas e busca o template sugerido, depois "Importar e Clonar" — deep clone das viaturas/itens para um cartão novo (não-template), com `comandante` zerado. Se não houver template compatível, o sistema avisa mas não trava. **Não há mais inferência automática do tipo pela data** (decisão 2026-07): a escolha manual sempre prevalece, permitindo preparar cartões de turnos futuros independentemente da data de criação. O `tipo_periodo` é gravado no cartão do dia (POST `/api/cartoes` lê do body; cartão clonado herda do template) e é editável no cabeçalho do cartão via `#cartao-tipo-periodo` (PUT `/api/cartoes/:id`, Adjunto pode).
- **Perfis de acesso:**
  - `P3` — acesso total (admin), incluindo Cadastro de Pessoal, Cadastro de Bairros e Templates do Cartão Programa.
  - `Adjunto` / `Oficial` — só veem Meu Turno, Cartão Programa, Listar Eventos e Mapa (leitura). Cartão Programa é a única coisa que eles podem *editar* (inclusive usar o fluxo de sugestão/clonagem de template — essa parte do endpoint de clonagem não exige P3 de propósito). A aba **Operações** é P3-only.
- Controle de acesso é **client-side (classe `hidden-role`) E server-side (middleware `exigirP3`)** — sempre implementar os dois lados, nunca confiar só no frontend.

## Segurança (já implementada — não regredir)

- Senhas com hash `scrypt` (nunca texto puro). Migração automática de senhas legadas roda no boot do servidor.
- Sessões via token Bearer, expiram em 12h, invalidadas no logout. Sessões expiradas de qualquer usuário são apagadas globalmente no boot do servidor (dentro do IIFE `inicializar()`), além da limpeza pontual das sessões expiradas do próprio usuário que já acontecia a cada login.
- Toda rota `/api/*` (exceto `/api/login`) exige token válido (middleware `autenticar`), que popula `req.user = { usuario, role, nome }`.
- Rotas de escrita administrativas exigem `exigirP3`. `POST /api/cartoes` faz a checagem de role manualmente dentro do handler (em vez de usar o middleware) só quando `is_template` é true, porque a mesma rota também cria o cartão do dia — que Adjunto/Oficial podem criar livremente — ler o handler antes de assumir que uma rota tem ou não restrição. `DELETE /api/cartoes/:id` usa o middleware `exigirP3` para qualquer cartão (template ou do dia) — apagar o roteiro operacional de um dia é destrutivo e não faz parte da edição diária que Adjunto/Oficial têm permissão de fazer no Cartão Programa.
- **Toda inserção de dado do usuário no DOM passa pela função `esc()`** (escape de HTML) em `app.js` — proteção contra XSS. Ao adicionar `innerHTML` novo, sempre envolver valores dinâmicos com `esc(...)`.
- Senha mínima: **3 caracteres** (decisão deliberada, para não quebrar a senha padrão `123` já usada pelo batalhão — não "corrigir" para 4+ sem perguntar).
- **RLS habilitado em todas as tabelas públicas** (migration `ativar_rls_tabelas_publicas`, 2026-07), **sem policies**. O `service_role` que o `server.js` usa tem `BYPASSRLS`, então RLS não afeta nenhuma rota do backend; sem policy, os roles `anon`/`authenticated` ficam totalmente bloqueados no acesso direto ao Supabase (o frontend não fala com o Supabase direto). Isso fecha o alerta `rls_disabled_in_public` do Supabase. Ao criar tabela nova, habilitar RLS nela também (`ALTER TABLE ... ENABLE ROW LEVEL SECURITY`, sem policy) — ver bloco no fim de `supabase/schema.sql`. **Não** criar policies sem necessidade real (quebraria o padrão de autorização feito no Express).

## Convenções de código

- **Idioma:** português em tudo — variáveis, funções, comentários, textos de UI. Não anglicizar.
- **Chamadas à API no frontend:** sempre via `apiFetch()` (wrapper que injeta o token e trata sessão expirada), nunca `fetch()` puro.
- **Handlers de clique: nada de `onclick=""` inline.** A CSP (`server.js`) tem `script-src` sem `'unsafe-inline'`, então handler inline **não dispara**. Todo botão renderizado dinamicamente usa `data-action="..."` (+ `data-*` para os argumentos, lidos via `el.dataset`) e é despachado por um **único listener delegado em `document`** (no bloco `DOMContentLoaded` de `app.js`, `switch (d.action)`). `document` é o pai estável — sobrevive ao `innerHTML` reescrito das listas. Ao criar um botão novo: adicionar o `case` no switch e passar os dados por `data-*` com `esc(...)` (nunca concatenar argumento em string inline — quebra com aspas/apóstrofo).
- **Modais:** padrão `.modal-overlay` + `.modal-box`, toggle via classe `hidden`. Ver `modal-usuario`, `modal-pessoa`, `modal-novo-template`, `modal-relatorio-sei` como referência.
- **Drawers/off-canvas:** três componentes distintos, não confundir — `#drawer` (gaveta de detalhes de **evento**, `openDrawer(eventId)`/`closeDrawer()`, só alocação agregada, sem escala); `#drawer-op` (gaveta de detalhes de **operação**, `openDrawerOperacao(id)`/`closeDrawerOperacao()`, com bloco "Efetivo Escalado" que reaproveita os mesmos ids do form de escala — `form-escala`/`escalas-list`/`esc_*` — que só existem nessa gaveta agora); `.sidebar`/`#nav-drawer-overlay` (classes `.nav-drawer-open`/`.open`, `abrirNavDrawer()`/`fecharNavDrawer()`) é o menu de navegação mobile, só ativo em `@media (max-width:768px)` — em desktop a sidebar continua fixa, sem drawer. Ambos usam a mesma técnica visual (wrapper fixo + overlay com fade + painel deslizando via `transition`), mas com nomes de classe/função próprios de cada um.
- **Painéis colapsáveis** (Histórico do Cartão, Templates, Gerenciar Bairros): botão toggle + `panel.classList.toggle('hidden')`, renderiza sob demanda só quando abre.
- **Autocomplete customizado** (form "Escalar Militar" na `#drawer-op`): o campo Nome (`#esc_militar_nome`, `.autocomplete-wrap`) filtra `state.pessoal` por nome/`nome_guerra`/`matricula` (via `normalizarTexto`, acento- e case-insensitive) e mostra um dropdown próprio (`#escala-autocomplete-results`, `.autocomplete-item`) — substitui o `<datalist>` nativo (ruim no mobile). Funções `renderAutocompleteEscala`/`selecionarMilitarEscala`/`limparAutocompleteEscala` em `app.js`; seleção via `data-action="selecionar-militar-escala"` no listener delegado; navegação por teclado (setas+Enter) e fecha ao clicar fora/Esc. Selecionar trava a matrícula (`readOnly`) só se o militar tiver matrícula; limpar o nome libera de novo (permite escalar militar fora do cadastro por digitação livre). Chamar `limparAutocompleteEscala()` em todo reset do `form-escala`. Padrão reaproveitável para outros autocompletes (classes `.autocomplete-*` genéricas em `style.css`).
- **Toasts:** `showToast(mensagem, tipo)` — tipos: `success`, `info`, `warning`, `danger`.
- **Badges de tipo/status:** classe CSS gerada via slug (minúsculo, espaço→hífen, remove acento com `normalize("NFD")`). Ao adicionar um novo tipo/atividade/categoria, lembrar de adicionar a cor correspondente em `style.css`. Exceção: as 5 categorias operacionais do Cadastro de Pessoal (`.badge.pcat-*`, variáveis `--badge-categoria-1..5`) usam um mapa fixo (`CATEGORIA_PESSOAL_BADGE_MAP` em `app.js`) em vez do slug automático, porque os nomes têm preposições/acentos que não convertem de forma previsível.
- **Gráficos:** sem lib externa — SVG gerado à mão em JS (ver `renderSazonalidadeChart` em `app.js`), consistente com o resto do projeto (mini-barras em CSS já usadas em Estatísticas/Planejador).
- **Mapa:** Leaflet via CDN, tiles dark (`basemaps.cartocdn.com/dark_all`), overrides de tema em `style.css` (`.leaflet-popup-*`, `.mapa-*`).
- **Toda funcionalidade nova** deve funcionar corretamente nos dois lados de sessão: o que o `P3` vê/edita e o que `Adjunto`/`Oficial` vê (geralmente menos).
- **Migrações de schema:** IIFE no topo do `server.js` (junto das outras, ex. `normalizarCamposTemplateCartao`, `seedBairrosCoordenadas`) que roda no boot, lê o db, adiciona campos faltantes com `if (!('campo' in obj))`, e só grava (`writeDB`) se algo mudou. Nunca destrutivo.

## Módulos principais (abas)

1. **Dashboard** (P3 apenas — `hidden-role` para Adjunto/Oficial) — visão geral, alertas operacionais consolidados (conflitos do Cartão Programa de hoje + eventos sem Número da OS/SEI se aproximando), preview do patrulhamento de hoje, e grid de cards-resumo clicáveis (um por módulo: Eventos, Cartão Programa, Planejador de Diárias, Relatório de Diárias, Estatísticas, Cadastro de Pessoal, Usuários) alimentado por uma única chamada a `GET /api/dashboard-resumo` (rota agregadora, `exigirP3`, evita múltiplas chamadas paralelas do frontend). Não existe mais o painel de calendário mensal do Dashboard — foi removido deliberadamente em favor do grid de cards; o calendário do Planejador de Diárias (`renderCalendarioDiarias`) é outro componente e não foi tocado.
2. **Novo Evento** — cadastro de eventos. Bairro é um `<select>` do cadastro + opção "Outro" (texto livre).
3. **Listar Eventos** — consulta geral com filtro de datas, colunas Nº OS / Nº SEI, e botão **"Relatório de Eventos (PDF)"** (ver seção "Relatórios PDF"): gera relatório no estilo SGEPM (colunas Nº · Data · Nº OS · Nome do Evento · Endereço/Local · Nº SEI) dos eventos do filtro ativo, via `getEventosFiltrados()`. Não tem mais coluna/indicador de diária (eventos não têm diária).
4. **Mapa** — eventos da semana corrente plotados por bairro (Leaflet dark mode), aviso quando o bairro do evento não está cadastrado. Painel "Gerenciar Bairros" (P3) para CRUD do cadastro de coordenadas.
5. **Meu Turno** — visão do dia para Adjunto/Oficial.
6. **Cartão Programa** — roteiro diário de patrulhamento por viatura: viaturas, itens de roteiro, Quadro Resumo automático, histórico navegável, alertas de conflito (sobreposição de horário / setor sem cobertura / Fiscal Praça sem Sobreaviso), impressão, **Templates** (criar/gerenciar/sugerir/clonar). Cada item de roteiro tem ação rápida **"Mudar atividade"** (troca o badge por `<select>` inline, salva via `PUT .../itens/:iid` com `{atividade}` — Adjunto pode; `editandoAtividadeItem` + `data-action` mudar/salvar/cancelar-atividade-item). Botão **"Copiar"** (antes "Copiar do Dia Anterior"): abre o modal `#modal-copiar-cartao` para o operador escolher **qualquer cartão existente** (qualquer data, exceto a alvo) como origem de uma cópia (deep clone de viaturas/itens) para a data-alvo; backend `POST /api/cartoes` com `copiar_de` = id do cartão de origem (o valor `'ultimo'` segue aceito por compatibilidade, mas não é mais acionado pela UI). A cópia herda o `tipo_periodo` da origem.
7. **Operações** (P3) — lista + gaveta (`#drawer-op`) das operações com diária: badge de `situacao`, coluna de diária (real ou estimada), "Efetivo Escalado" editável, "Marcar como Executada", exclusão com confirmação reforçada (mostra militares/diárias que serão perdidos), Relatório SEI. Modal Nova/Editar Operação. `/api/operacoes` (CRUD) + `/api/escalas?operacao_id=`.
8. **Relatório Diárias** — (a) consolidado por militar (fonte: operações+escalas), exportável em CSV **e em PDF estilo SGEPM** (`gerarRelatorioPdfConsolidado`, ver seção "Relatórios PDF"); (b) **Relatório Diário** (`GET /api/relatorio-diario?mes&ano&agrupar=data|operacao`, `exigirP3`): painel com toggle Por Data / Por Operação e botão **"Relatório (PDF)"** (estilo SGEPM, ver seção "Relatórios PDF") — tabela agrupada por data/operação com total por grupo + total do mês. Grad abreviado via `ABREV_POSTO` (app.js). Resolve posto+nome de guerra cruzando `escala.militar_id` → `pessoal.matricula` (fallback `militar_nome`). Usa `escala.total_diarias` (não recalcula). Agregação por data usa a `data_inicio` da operação (escalas não têm data própria).
9. **Planejador Diárias** — cota mensal x consumo, calendário de diárias, lançamento rápido de "Missão Avulsa" (que agora cria uma **operação** Planejada, não um evento). Lista as operações do mês; **não há mais tabela separada de Missões Planejadas** (viraram operações).
10. **Estatísticas** — painel analítico (eventos: por bairro/tipo/modalidade; diárias: de operações+escalas; patrulhamento: por setor/atividade/viatura), sempre com filtro de ano.
11. **Usuários** — gestão de contas de login do sistema, reset de senha (só P3).
12. **Cadastro de Pessoal** — Adjuntos, Fiscais de Operações, Oficiais de Operações, Oficiais de Sobreaviso, com filtro por categoria (só P3).

## Relatórios PDF (estilo SGEPM)

Os antigos botões de texto "SEI" (Gerar Relatório para SEI / Lista para SEI em Listar Eventos, Relatório SEI nas gavetas de evento/operação, Copiar Texto Formatado no Relatório Diário) foram **substituídos por relatórios em PDF** inspirados no relatório oficial SGEPM (cabeçalho azul-marinho `#002145`, tabela zebrada `#F5F7FA`, A4 paisagem, Helvetica), gerados via `window.print()` (Salvar como PDF). **Todos os relatórios PDF compartilham o mesmo design** (é requisito do usuário — nenhum relatório pode ficar fora do padrão). Três relatórios:
- **"Relatório de Eventos (PDF)"** (Listar Eventos) — `gerarRelatorioPdfEventos()`, colunas Nº · Data · Nº OS · Nome do Evento · Endereço/Local · Nº SEI, respeitando o filtro ativo (`getEventosFiltrados()`).
- **"Relatório (PDF)"** (Relatório Diárias → Relatório Diário) — `gerarRelatorioPdfDiario()`, tabela agrupada por data/operação com total por grupo + total do mês, grad abreviado.
- **"Relatório (PDF)"** (Relatório Diárias → Consolidado por Militar) — `gerarRelatorioPdfConsolidado()`, colunas Nº · Matrícula · Nome do Militar · Escalas · Aparições · Total de Diárias, respeitando mês/ano + filtro de busca, com rodapé de total de militares/diárias. Fonte: `/api/relatorio-diarias`. O botão "Exportar" (CSV) do mesmo painel foi **mantido** (export de planilha é função à parte do relatório impresso).

Os três injetam HTML em `#relatorio-pdf-area` (dentro do modal único `#modal-relatorio-pdf`) via `abrirRelatorioPdf()` + `cabecalhoRelatorioPdf(titulo, sub)` (+ `.rel-pdf-tabela` e, quando há total simples, `.rel-pdf-rodape`). **Mudança de design de relatório vale para todos:** alterar `cabecalhoRelatorioPdf` e/ou as classes `.rel-pdf-*` (compartilhadas) já propaga para os três — só tratar um relatório isoladamente se o usuário pedir explicitamente. Ao criar um relatório PDF novo, reusar esses helpers/classes, nunca montar HTML/CSS do zero. Impressão isolada com o mesmo padrão do antigo modal SEI: `@media print { body:has(#modal-relatorio-pdf:not(.hidden)) ... }` esconde `.app-container` e mostra só a área, com `@page relatorioPdfLandscape { size: A4 landscape }` e `print-color-adjust: exact` (para o cabeçalho marinho/zebra saírem coloridos).

**Código SEI legado removido:** os modais `#modal-relatorio-sei` e `#modal-lista-sei`, as funções `abrirRelatorioSei`/`renderRelatorioSei`/`montarTextoRelatorioSei`/`handleCopiarRelatorioSei`/`abrirModalListaSei`/`montarTextoListaSei`/`handleCopiarListaSei`, a rota `GET /api/relatorio-sei` (+ helper `consolidarEfetivoSei`) e o CSS `.sei-*` foram **apagados** (não existem mais). Preservado: a classe `.lista-sei-texto` (ainda estiliza o `#rel-diario-texto` do Relatório Diário on-screen) e `getEventosFiltrados`.

## Fluxo de trabalho recomendado ao editar

1. Ler o trecho relevante antes de editar (arquivos grandes — usar Grep para localizar, não abrir tudo).
2. Depois de editar `.js`, rodar `node --check <arquivo>` antes de testar no navegador.
3. Reiniciar o preview server (`server.js` não tem hot reload).
4. Testar de ponta a ponta no navegador (login, ação, verificação visual) — não declarar "pronto" sem isso. Testar com clique real (preview_click/preview_fill), não só chamando a função handler direto por `preview_eval` — já aconteceu de um fluxo funcionar via chamada direta e mascarar um problema de timing no clique real.
5. **Sempre limpar dados de teste** criados durante a verificação (eventos, cartões, templates, pessoal, bairros fictícios) antes de encerrar — o preview local aponta pro mesmo Supabase de produção (mesmas credenciais em `.env.local`/`.claude/launch.json`), não é um ambiente descartável. Exceção: quando o próprio usuário pede explicitamente para popular dados fictícios/demo — aí é pra manter.
6. Testar nos dois perfis relevantes quando a mudança afeta permissões (P3 e Adjunto).
7. Antes de mexer ou apagar qualquer registro no Supabase, checar se os dados batem com o que você mesmo criou nesta sessão — pode haver uso real do app em paralelo (outro servidor do usuário rodando, ou a própria produção na Vercel). Se encontrar registro que você não reconhece ter criado, não delete: avise o usuário.
8. Depois de validar localmente, `git add`/`commit`/`push` para `main` — a Vercel publica automaticamente via integração Git (ver seção "Stack e arquitetura"). Não fazer push sem o usuário saber que aquilo vai pra produção.

## Pendências em aberto 

Nenhuma pendência de arquitetura estrutural em aberto no momento. Cadastro de Viaturas/Frota já existe (ver seção "Módulos principais"), e o Cadastro de Pessoal já foi populado com o efetivo geral do batalhão (244 militares importados do relatório SGEPM em 2026-07, com campo `matricula`) — a maioria sem `categorias` (efetivo geral sem papel operacional ainda; ver nota em "Modelo de dados").

**Itens de auditoria de segurança/performance adiados (2026-07):** uma auditoria aplicou correções em 4 lotes (backup sem senha/sessão + leituras pontuais; trust proxy + senha mínima de 8; sanitização de badge + `tipo_evento` fechado + indexação das agregações; SRI nos CDNs + polling pausado com aba oculta). Ficaram **deliberadamente fora de escopo** desta rodada, para fases próprias:
- **S1 — CORS sem `Origin`:** `origemPermitida` devolve `true` quando não há header `Origin` (curl/apps nativos). Comportamento mantido de propósito; a autorização real é por token, não por CORS.
- **S2 — token em `localStorage`:** migrar o token de sessão de `localStorage` para cookie `HttpOnly` é mudança estrutural de auth (fase própria).
- **S4 — `'unsafe-inline'` no `style-src`:** só sai depois de migrar dezenas de `style=` inline para classes (junto da fase de design/Plano 3).
- **P2 — aposentar `writeDB` de vez:** as rotas de escrita de tabela única já usam `writeRow`/`readTabela`, mas as agregadoras e as demais ainda usam o shim `readDB()`/`writeDB()`. Substituição total fica para a fase de performance.
- **P5 — índices/paginação no Supabase:** sem índices dedicados nem paginação nas listagens grandes; fase de performance.
- **S10 — confirmar RLS em produção:** verificar via `get_advisors` (Supabase MCP) que o `rls_disabled_in_public` está fechado — exige aprovação manual do tool call, apenas reportar, sem alterar nada.

**Separação Eventos/Operações (concluída em 2026-07):** a diária saiu dos eventos e passou para a nova entidade `operacoes` (registro único planejamento→execução). A antiga `missoes_planejadas` foi migrada para `operacoes` e a tabela ficou **órfã no banco** (dados migrados, sem `DROP` — rede de segurança, mesmo padrão da antiga tabela de auditoria; não recriar dependência nela). O `data/db.json` legado segue preservado. Se algum dia for dar `DROP TABLE missoes_planejadas`, confirmar antes com o usuário.

Ideias sugeridas e ainda não priorizadas pelo usuário: Setores de Patrulhamento com vínculo N:N a Bairros, Demandantes, Modalidades de Policiamento, para eliminar texto livre restante em outras telas (demandante do evento, modalidade de policiamento).
